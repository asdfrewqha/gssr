# Step 12: k3s Cluster (Optional — Kubernetes on Zynq)

## Goal

Set up a lightweight Kubernetes cluster with k3s: control plane on Main PC, worker nodes on all 5 Zynq ARM nodes. Enables rolling updates, health checks, and proper orchestration.

## When to Use This Step

Use k3s **only if** your Zynq nodes run proper Linux (Debian, Ubuntu, or Armbian) with:

- At least 512 MB RAM per node
- cgroups v1 or v2 support (`cat /proc/cgroups`)
- Kernel 5.x+

If nodes run OpenWrt or have less RAM, use **Step 11 (Docker or bare binary)** instead.

## Prerequisites

- Step 3 (game service Docker image) published to DockerHub
- All Zynq nodes SSH-accessible from Main PC
- Zynq nodes have proper Linux (not OpenWrt)

## Steps

### 1. Install k3s server on Main PC

```bash
curl -sfL https://get.k3s.io | sh -
# k3s server starts automatically

# Get node join token
sudo cat /var/lib/rancher/k3s/server/node-token
# Save this as NODE_TOKEN
```

### 2. Install k3s agent on each Zynq

```bash
# Run on each Zynq node:
export MAINPC_IP="192.168.1.100"
export NODE_TOKEN="<token-from-step-1>"

curl -sfL https://get.k3s.io | \
  K3S_URL="https://${MAINPC_IP}:6443" \
  K3S_TOKEN="${NODE_TOKEN}" \
  INSTALL_K3S_EXEC="--node-label role=game" \
  sh -
```

### 3. Verify all nodes joined

```bash
# On Main PC:
sudo kubectl get nodes
# Should show: mainpc (control-plane) + zynq1..5 (worker, role=game)
```

### 4. Export kubeconfig for use without sudo

```bash
sudo cp /etc/rancher/k3s/k3s.yaml ~/.kube/config
sudo chown $USER ~/.kube/config
kubectl get nodes   # works without sudo
```

### 5. Deploy the game service

```bash
kubectl apply -k infra/k8s/
# Creates: namespace/gssr, deployment/game, service/game, configmap/game-config
```

### 6. Check rollout

```bash
kubectl rollout status deployment/game -n gssr
kubectl get pods -n gssr -o wide
# 5 pods, each on a different Zynq node
```

### 7. Update Nginx upstream

When using k3s, Nginx uses the k3s NodePort service instead of individual node IPs:

```nginx
upstream zynq_game {
    server <MAINPC_IP>:<nodeport>;   # k3s NodePort, e.g. 30000
}
# kube-proxy load-balances across all 5 pods internally
```

Or use a ClusterIP + Nginx inside the cluster (advanced).

## Rolling Updates

To deploy a new image version:

```bash
kubectl set image deployment/game \
  game=user/gssr-game:$NEW_SHA \
  -n gssr

kubectl rollout status deployment/game -n gssr --timeout=120s
# → "successfully rolled out"
```

k3s performs a rolling update: brings up 1 new pod, waits for readiness, then terminates 1 old pod. Zero downtime.

## CI/CD Integration

Add secret `K3S_KUBECONFIG` to GitHub (contents of `~/.kube/config`).

In `.github/workflows/deploy-k3s.yml`:

```yaml
- name: Deploy new image
  run: |
    echo "${{ secrets.K3S_KUBECONFIG }}" > /tmp/kubeconfig
    kubectl --kubeconfig /tmp/kubeconfig \
      set image deployment/game game=user/gssr-game:${{ github.sha }} -n gssr
    kubectl --kubeconfig /tmp/kubeconfig \
      rollout status deployment/game -n gssr --timeout=120s
```

## Useful kubectl Commands

```bash
# Watch pods
kubectl get pods -n gssr -w

# Logs from all game pods
kubectl logs -n gssr -l app=game --all-containers

# Scale down (e.g., maintenance)
kubectl scale deployment/game --replicas=0 -n gssr

# Shell into a pod
kubectl exec -it -n gssr <pod-name> -- /bin/sh

# Delete and recreate all pods (force restart)
kubectl rollout restart deployment/game -n gssr
```

## Monitoring with Promtail DaemonSet

Instead of per-node Promtail, deploy it as a DaemonSet inside k3s:

```bash
kubectl apply -f infra/k8s/monitoring/promtail-daemonset.yaml
# Collects logs from all k3s pods, ships to Loki on Main PC
```

## Verification

1. `kubectl get nodes` — all 5 Zynq nodes show `Ready`
2. `kubectl get pods -n gssr` — 5 game pods, all `Running`
3. `curl https://game.school.example.com/health` — responds correctly
4. Deploy new image → `kubectl rollout status` completes, no downtime

## Troubleshooting

- **Zynq fails to join cluster**: check firewall — Main PC must allow TCP 6443 from Zynq subnet
- **Pods stuck in `Pending`**: check node labels (`kubectl get nodes --show-labels`) and affinity rules
- **"ImagePullBackOff"**: DockerHub image is private or wrong tag — check imagePullSecrets
- **cgroups error on agent start**: enable cgroups on Zynq: add `cgroup_enable=memory cgroup_memory=1` to `/boot/cmdline.txt` and reboot
